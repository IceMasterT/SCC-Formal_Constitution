# Reproducibility card

## Evaluation unit

One candidate runtime transition: previous state, candidate next state, canonical event, audit event, proof-obligation bundle, and execution descriptor.

## Determinism assumptions

The checker decision must not depend on wall-clock time, randomness, network calls, environment variables, telemetry, mutable globals, or product-runtime decisions.

## Positive controls

- `golden/valid_exact.json`
- `golden/valid_stutter.json`
- `golden/valid_safe_refined.json`

Each positive control has JSON carrier form and committed binary canonical vectors under `golden/bin/`.

## Negative controls

The negative corpus contains fourteen committed fixtures. See `docs/fixture_gate_map.md` and `negative_corpus/manifest.json`.

## Release-blocking commands

```bash
python3 tools/validate_vectors.py
python3 tools/forbidden_imports.py
cd rust && cargo test --all-features
cd rust && cargo test --test negative_corpus
cd rust && cargo test --test golden_vectors
cd rust && cargo test --test proptest_invariants
cd ts && npm test
```

## Evidence policy

The paper and README intentionally separate observed evidence from required release gates. A local sandbox without Rust may validate vectors and TypeScript behavior, but it cannot be used to claim that the Rust source-of-truth checker passed. For submission or camera-ready release, attach the transcript generated by `scripts/collect_ci_evidence.sh` on a Rust-equipped runner.
